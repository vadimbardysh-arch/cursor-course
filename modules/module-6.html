Модуль 6

🐙 GitHub — автоматизація та публікація

Як зберігати файли, публікувати звіти як вебсайти та автоматизувати рутину — без жодного рядка коду вручну

⏱ 20 хвилин 📝 5 питань 🎯 Для початківців
📖
На нашій кухні це: GitHub — це книга рецептів + вітрина ресторану. Зберігає всі рецепти (код), а вітрина (GitHub Pages) показує готові страви перехожим. GitHub Actions — це автоматична лінія, яка готує за розкладом без шеф-повара.

🤔 Що таке GitHub?

Ти знаєш Google Drive — хмарне сховище, куди можна закидати файли й ділитися ними з колегами. А тепер уяви, що Google Drive отримав суперсили:

💡 GitHub = Google Drive + повна історія змін + автоматичні задачі + безкоштовний хостинг вебсайтів. Все в одному!

Головне, що потрібно зрозуміти: нам НЕ потрібно писати код, щоб використовувати GitHub. Cursor робить усе за нас. Ми лише говоримо, що хочемо, а він виконує.

Три речі, для яких ми використовуємо GitHub:

📁

Зберігання файлів та звітів

Усі наші скрипти, звіти та дані зберігаються на GitHub — надійно, з історією змін, доступно з будь-якого комп'ютера

🌐

Публікація звітів як вебсайтів

GitHub Pages перетворює твій HTML-звіт на справжній вебсайт з URL-адресою. Надіслав посилання партнеру — він бачить красивий звіт у браузері

🤖

Запуск автоматичних задач

GitHub Actions — це робот, який виконує завдання за розкладом. Наприклад, щопонеділка генерує свіжий звіт

📖 Ключові поняття (дуже просто!)

Не лякайся термінів — зараз усе розкладемо по поличках. Уяви, що GitHub — це Google Drive на стероїдах:

📂 Repository (репозиторій, або просто «репо»)

Це папка проєкту на GitHub. Як папка в Google Drive — тільки з суперсилами. Один проєкт = один репозиторій.

Приклад

У тебе є папка «partner-reports» — це і є твій репозиторій. Там лежать звіти, скрипти, стилі.

💾 Commit (коміт)

Це збереження «знімку» всіх файлів у певний момент. Як натиснути Cmd + S, але з магією — GitHub запам'ятовує ВСІ попередні версії. Зламав щось? Завжди можна відкотитися назад.

Аналогія

Уяви, що ти робиш фото робочого столу кожного разу, як щось змінюєш. Потім можеш подивитися будь-яке попереднє фото і повернути все як було.

⬆️ Push (пуш)

Це завантаження змін на GitHub. Ти зробив коміт у себе на комп'ютері — а пуш відправляє його в хмару. Як синхронізація Google Drive.

🌿 Branch (гілка)

Це копія проєкту, де можна експериментувати, не ламаючи оригінал. Як чернетка документа — пишеш, тестуєш, а коли все ок — зливаєш із основним файлом.

👑 Main / Master

Це головна «офіційна» версія проєкту. Все, що тут — працює і перевірене. Гілки створюються саме від main.

✅ Запам'ятай формулу: Зробив зміни → Commit (зберіг знімок) → Push (відправив у хмару). Три кроки — і все на GitHub!

🔧 Початкове налаштування (крок за кроком)

Щоб почати працювати з GitHub, потрібно зробити налаштування один раз. Після цього все працюватиме автоматично.

1

Створи акаунт на GitHub

Зайди на github.com і зареєструйся. Це безкоштовно. Запам'ятай свій логін — він буде частиною URL твоїх сайтів.

2

Встанови GitHub CLI

GitHub CLI (gh) — це інструмент для роботи з GitHub через термінал. Cursor може встановити його за тебе! Просто напиши в чат:

Встанови GitHub CLI (gh) на мій Mac
3

Авторизуйся в GitHub

Попроси Cursor:

Авторизуй мене в GitHub через gh auth login
4

Підтверди в браузері

Cursor покаже тобі код і посилання. Відкрий посилання в браузері, введи код — готово! Тепер Cursor має доступ до твого GitHub.

💡 Це потрібно зробити лише один раз! Після налаштування Cursor пам'ятатиме твій акаунт і працюватиме з GitHub автоматично.

🌐 GitHub Pages — твій безкоштовний вебсайт

Це одна з найкрутіших фішок GitHub. Уяви: ти створив HTML-звіт у Cursor — і хочеш, щоб партнер побачив його у браузері за посиланням, без скачування файлів. GitHub Pages робить саме це!

💡 GitHub Pages — це безкоштовний хостинг від GitHub. Він бере твої HTML-файли і перетворює їх на справжній вебсайт з адресою типу https://твій-логін.github.io/назва-репо/

Чому це класно:

✨ Безкоштовно — ніяких платних хостингів

✨ Автоматично — пушнув зміни → сайт оновився

✨ Публічне посилання — надіслав URL партнеру, він відкрив і бачить звіт

✨ Красиво — повноцінний вебсайт з графіками, таблицями, брендингом

Як налаштувати (крок за кроком):

1

Попроси Cursor створити репозиторій

Напиши в чат:

Створи новий GitHub репозиторій partner-reports з GitHub Pages
2

Cursor все налаштує

Він створить репозиторій на GitHub, увімкне Pages і підготує структуру папок. Тобі нічого не потрібно робити!

3

Твій сайт готовий!

Звіти тепер доступні за адресою:
https://твій-логін.github.io/partner-reports/

4

Оновлення — автоматичні

Кожного разу, коли ти робиш push — сайт автоматично оновлюється. Партнер бачить завжди свіжу версію за тим самим посиланням!

✅ Підсумок: Ти більше не надсилаєш файли поштою. Просто даєш посилання — і партнер бачить завжди актуальний звіт у браузері. Професійно і зручно!

🤖 GitHub Actions — автоматичні задачі

А тепер уяви таку ситуацію:

Сценарій

Кожного понеділка о 13:00 тобі потрібно: зайти в Databricks, витягти свіжі дані, згенерувати звіт, опублікувати його на сайті. Це займає 30 хвилин ручної роботи.

А тепер уяви, що ти наймаєш робота, який робить усе це за тебе. Щопонеділка о 13:00 він запускається, виконує всі кроки і публікує звіт — а ти в цей час п'єш каву ☕

Це і є GitHub Actions — автоматичні процеси (workflows), які запускаються за розкладом або за подією.

Що ми автоматизуємо:

📊

Автогенерація звітів щотижня

Скрипт запускається за розкладом, збирає дані, генерує HTML-звіт

🗄️

Автоматичне витягування даних з Databricks

Підключається до бази, виконує SQL-запити, отримує свіжі цифри

🌐

Оновлення GitHub Pages

Після генерації звіту — автоматично пушить його на сайт. Партнер бачить свіжий звіт без твоєї участі

⚙️ Як працюють GitHub Actions (спрощено)

Кожна автоматична задача описується у спеціальному файлі — YAML-файлі (це як рецепт для робота). Цей файл лежить у папці .github/workflows/.

⚠️ Але! Тобі НЕ потрібно писати цей файл самостійно. Cursor зробить це за тебе! Просто опиши, що хочеш автоматизувати.
Приклад промпту для Cursor
Створи GitHub Action, яка кожного понеділка о 13:00
запускає скрипт generate_report.py
і пушить результат на GitHub Pages

Cursor створить workflow-файл, який містить три ключові частини:

⏰

КОЛИ запускати (schedule)

Кожного понеділка о 13:00 — або за іншим розкладом, який ти вкажеш

📋

ЩО робити (steps)

Встановити залежності → Запустити скрипт → Зібрати результат → Запушити на GitHub Pages

☁️

ДЕ виконувати

На серверах GitHub! Не на твоєму комп'ютері. Ти можеш навіть не вмикати ноутбук — робот працює в хмарі

💡 Аналогія: GitHub Actions — це як замовити доставку їжі через додаток. Ти один раз кажеш «кожного понеділка привози мені піцу о 13:00» — і далі вона приїжджає автоматично. Тобі не потрібно розуміти, як працює кухня чи логістика.

🔄 Повний робочий процес

Давай подивимось, як усі частини складаються в одну систему:

1

Databricks

Тут живуть наші дані — замовлення, партнери, метрики

2

Скрипт (Python)

Cursor написав скрипт, який підключається до Databricks, витягує дані й генерує HTML-звіт

3

GitHub

Скрипт і звіт зберігаються в репозиторії на GitHub

4

GitHub Pages

HTML-звіт автоматично стає вебсайтом з публічним URL

5

GitHub Actions

Кожного тижня робот запускає скрипт → генерує новий звіт → публікує на Pages

❌ Без автоматизації:

Кожного понеділка вручну: відкрий Databricks → напиши запит → скопіюй дані → зроби звіт → відправ партнеру → повтори 10 разів для різних партнерів

✅ З GitHub + Actions:

Один раз налаштував → далі робот щотижня генерує звіти → партнери бачать свіжі дані за постійним посиланням → ти п'єш каву ☕

✅ Формула успіху: Databricks (дані) → Скрипт (обробка) → GitHub (сховище) → Pages (публікація) → Actions (автоматизація) → ☕ (ти відпочиваєш)
